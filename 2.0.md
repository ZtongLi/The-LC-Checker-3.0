### 3.3 [文件 3] `agent_enhance/planner_agent.py` — 规划 Agent

**职责**：在正式 AI 审核前，扫描所有 4 份单据的结构化字段数据，识别其中的专业术语和歧义表达，决定后续哪些术语需要做知识增强。

**对应 KDR-Agent**：`tool.py` 的 `get_palnner_prompt()` (L12-40) + `main.py` Stage 1 (L43-46)

**详细设计**：

| 方法 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `__init__(llm_client)` | AgentLLMClient | 无 | 注入 LLM 客户端 |
| `analyze(lc, invoice, bl, insurance)` | 4 个 dict | `PlannerResult` | 分析全部单据字段 |
| `_build_field_summary(lc, invoice, bl, insurance)` | 4 个 dict | str | 将 4 份单据关键字段拼接为摘要文本 |
| `_parse_planner_output(answer)` | str | `PlannerResult` | 解析 LLM 回答中的术语列表 |

**`PlannerResult` 数据结构**：
```python
@dataclass
class PlannerResult:
    unfamiliar_terms: list[str]    # 陌生术语（如 "FPA", "ISBP 745 Para C13"）
    ambiguous_terms: list[str]     # 歧义表达（如 "approximately", "about"）
    key_risk_fields: list[str]     # 关键风险字段（如 "险别缺失War Risk"）
```

**输入来源**：直接接收 `server.py` L94-121 中已经提取好的 `lc_data`, `inv_data`, `bl_data`, `ins_data`。

**Planner 具体检查哪些内容**：
1. **信用证 `goods_description`** 中的贸易术语（Incoterms）、特殊产品名称
2. **保险单 `coverage`** 中的险别名称（ICC(A)、FPA、WPA、War Risk 等）
3. **提单** 中的特殊批注（charter party、on deck、shipper's load）
4. **跨单据** 名称差异（beneficiary vs shipper vs insured 三者是否存在拼写/缩写差异）

**为什么 LC-Checker 场景需要 Planner**：
- 信用证审核中 LLM 最容易出错的场景是"看不懂专业术语就瞎猜"
- 例：保险单写 `ICC(C) including TPND`，如果 LLM 不理解 TPND 是什么，就无法判断是否满足信用证要求的 "Theft, Pilferage and Non-Delivery"
- Planner 先识别出这些术语，交给 Knowledge Agent 查询，再注入审核 prompt

### 3.4 [文件 4] `agent_enhance/knowledge_agent.py` — 知识增强 Agent

**职责**：对 Planner 识别出的陌生术语和歧义表达，分别通过 **ChromaDB 知识库检索** 和 **LLM 上下文消歧** 获取准确解释，汇总为一段"背景知识摘要"。

**对应 KDR-Agent**：`tool.py` 的 `get_wiki_summary()` (L56-99) + `get_disambiguation_prompt()` (L43-54)

**详细设计**：

| 方法 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `__init__(llm_client, embed_model, collection)` | 三个对象 | 无 | 注入 LLM 客户端 + 现有的 ChromaDB 组件 |
| `enhance(planner_result, doc_context)` | PlannerResult, str | str | 完整知识增强流程，返回格式化的知识上下文 |
| `_retrieve_term_knowledge(term)` | str | str | 从 ChromaDB 检索术语对应的 UCP600 条款 |
| `_disambiguate_terms(terms, doc_context)` | list, str | str | 调用 LLM 对歧义表达做上下文消歧 |
| `_build_knowledge_context(...)` | 各种中间结果 | str | 将所有知识拼接成最终的上下文字符串 |

**与现有系统的关键连接点**：

```python
# 复用 check_v2.py 中已加载的 ChromaDB 组件（L17, L25-26）
# 不修改 check_v2.py，仅 import 使用：
from check_v2 import embed_model, collection
```

这是本方案的**关键设计决策**——Knowledge Agent 不需要自己初始化 ChromaDB，直接复用 `check_v2.py` 模块级已加载的 `embed_model` 和 `collection` 对象。因为 `check_v2.py` 在被 import 时（L17, L25-26）就会执行模块级代码完成初始化。

**知识增强的输出示例**：

```
【审核前知识增强 — 由 Knowledge Agent 提供】

术语解释：
1. "FPA" = Free from Particular Average（平安险），仅承保全损和共同海损，
   不承保单独海损。参见 UCP600 Article 28 关于保险单据的规定。
2. "CIF Incoterms 2020" = 成本+保险+运费，卖方负责运费和保险至目的港。
   据此，保险单应由卖方（受益人）出具。

上下文消歧：
1. "approximately 5,000 MT" — 根据 UCP600 Article 30(a)，使用 "approximately"
   一词允许数量有 ±10% 的偏差。因此 4,500-5,500 MT 均可接受。
2. "Co., Ltd." vs "Company Limited" — 根据 ISBP 745 惯例，这属于同一名称的
   等价缩写形式，不构成不符点。

重点风险提示：
- 信用证要求 "All Risks + War Risk"，保险单仅显示 "ICC(A)"。
  需要确认 ICC(A) 是否等价于 All Risks，以及 War Risk 是否单独承保。
```

### 3.5 [文件 5] `agent_enhance/reflection_agent.py` — 反思纠错 Agent

**职责**：接收 Phase 3+4 合并后的不符点列表，让 LLM 以"审核专家复核"的角色，按 4 个维度逐项检查，输出修正后的最终不符点列表。

**对应 KDR-Agent**：`tool.py` 的 `get_reflection_prompt()` (L163-191) + `main.py` Stage 3 (L106-132)

**详细设计**：

| 方法 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `__init__(llm_client, output_checker)` | 两个对象 | 无 | 注入 LLM 客户端和输出校验器 |
| `reflect(discrepancies, lc, invoice, bl, insurance, presentation_date)` | list + 4 dict + str | list | 完整反思流程 |
| `_build_reflection_prompt(discrepancies, lc, invoice, bl, insurance)` | list + 4 dict | str | 构建反思 prompt |

**4 维度检查标准**（对应 KDR-Agent 的 4 Checking Criteria）：

| KDR-Agent 原维度 | 迁移到 LC-Checker 的对应维度 | 检查内容 |
|-----------------|---------------------------|---------|
| 实体边界是否准确 | **字段匹配准确性** | 不符点涉及的信用证字段值和单据字段值是否真的存在差异？是否是 OCR 提取错误导致的误判？ |
| 实体类型是否准确 | **UCP 条款引用正确性** | 引用的 UCP600 条款是否真的适用于该不符点？例如保险金额问题应引用 Article 28(f)(ii) 而非 Article 18 |
| 边界+类型同时准确 | **严重程度合理性** | "严重/一般/轻微"的判定是否合理？缩写差异是否该降为轻微？金额超限是否确实严重？ |
| 是否遗漏实体 | **遗漏检查** | 是否有应检未检的不符点？特别是保险起运地/目的地、单据份数等容易被遗漏的项目 |

### 3.6 [文件 6] `agent_enhance/output_checker.py` — 结构化输出校验器

**职责**：对 LLM 的每次输出进行严格的 JSON Schema 校验，校验失败则触发重试。

**对应 KDR-Agent**：`jsonchecker.py` 的 `JsonChecker` 类 + `main.py` 中的 `answer_checker.check()` (L86-88, L110-112)

**详细设计**：

| 类/方法 | 说明 |
|--------|------|
| `DiscrepancySchema` | 定义不符点 JSON 对象必须包含的字段和类型 |
| `ReflectionSchema` | 定义反思输出 JSON 对象必须包含的字段和类型 |
| `OutputChecker.__init__(schema)` | 接收 Schema 定义 |
| `OutputChecker.check(data)` → bool | 校验单个 JSON 对象是否符合 Schema |
| `OutputChecker.check_list(data_list)` → bool | 校验 JSON 数组中每个元素是否都符合 Schema |
| `OutputChecker.parse_and_validate(raw_text, max_loop, llm_client, prompt)` → list | 解析文本→校验→不通过则用 LLM 重新���成→直到通过或达到上限 |

**Schema 定义对比**：

```python
# KDR-Agent 的 Schema（main.py L20-22）：
answer_schema = {"sentence": str, "entities": list}
answer_inner_schema = {"name": str, "type": str}

# LC-Checker V3 的 Schema：
DISCREPANCY_SCHEMA = {
    "document": str,        # 必填：涉及单据名称
    "field": str,           # 必填：不符的字段
    "description": str,     # 必填：不符点说明
    "ucp_reference": str,   # 必填：UCP600 条款引用
    "severity": str,        # 必填：严重/一般/轻微
    # 可选字段：lc_requirement, doc_shows, uncertain
}

REFLECTION_SCHEMA = {
    "action": str,          # 必填：keep/remove/modify/add
    # 根据 action 不同，其余字段有不同要求
}
```

**解决的核心问题**：现有 `check_v2.py` L494-514 中，`json.loads` 失败后只做 `re.search(r'\[.*\]', answer, re.DOTALL)` 的简单正则提取，经常丢失数据或解析错误。新的 `OutputChecker` 引入了：
1. `json_repair` 库容忍轻微格式问题（如尾逗号、单引号）
2. 字段级 Schema 校验（不只是"是不是合法JSON"）
3. max_loop 重试机制确保最终获得合规输出

### 3.7 [文件 7] `agent_enhance/contrastive_examples.py` — 正反对比样本管理器

**职责**：从 `contrastive_data/` 加载正反例，根据当前审核涉及的具体字段动态选择最相关的样本子集，格式化后返回可直接注入 prompt 的文本。

**对应 KDR-Agent**：`data/{dataset}/define_sample_min.json` 中含 `entities` + `false_entities` 的正反对比结构

**详细设计**：

| 方法 | 输入 | 输出 | 说明 |
|------|------|------|------|
| `__init__(data_dir)` | str (路径) | 无 | 加载全部 3 个 JSON 文件 |
| `select_relevant(lc, invoice, bl, insurance)` | 4 dict | `SelectedExamples` | 根据单据内容智能选择相关的正反例 |
| `format_for_prompt(selected)` | SelectedExamples | str | 将选出的正反例格式化为 prompt 文本 |

**智能选择逻辑**：

```python
def select_relevant(self, lc, invoice, bl, insurance):
    """根据当前单据内容，选择最相关的正反例子集"""
    selected = SelectedExamples()

    # 1. 如果存在金额容差，选择容差相关的案例
    if lc.get("amount_tolerance") and lc["amount_tolerance"] != "+/- 0%":
        selected.add_from_category("amount_tolerance")

    # 2. 如果受益人名称在各单据间不完全一致，选择名称类案例
    names = [lc.get("beneficiary", {}).get("name", ""),
             invoice.get("issued_by", ""),
             bl.get("shipper", ""),
             insurance.get("insured", "")]
    if len(set(n.lower().strip() for n in names if n)) > 1:
        selected.add_from_category("name_equivalence")

    # 3. 如果信用证有 "approximately" / "about" 等约数词
    goods_desc = lc.get("goods_description", "")
    if any(word in goods_desc.lower() for word in ["approximately", "about", "circa"]):
        selected.add_from_category("approximate_quantity")

    # 4. 如果涉及特定保险险别
    coverage = insurance.get("coverage", "")
    if coverage:
        selected.add_from_category("insurance_coverage")

    # 5. 始终包含通用港口/日期/缩写案例（高频误判）
    selected.add_from_category("port_matching")
    selected.add_from_category("general_abbreviations")

    return selected
```

### 3.8 [文件 8-13] `agent_enhance/prompts/` — Prompt 模板集

所有 Prompt 均抽离为独立的 Python 文件，使用函数返回格式化后的 prompt 字符串。这对应 KDR-Agent 中 `tool.py` 里的 `get_palnner_prompt()`、`get_disambiguation_prompt()`、`get_reflection_prompt()` 等函数。

### 3.11 [文件 18] `enhanced_review.py` — 增强版审核编排器（核心）

**职责**：这是整个 V3 系统的**唯一集成入口**，负责编排 Phase 0-6 的完整流程。它 import 现有模块的函数但不修改它们，同时调用所有新增的 Agent。

**详细设计**：

```
函数签名：
enhanced_review(
    lc_data: dict,          # 信用证结构化数据（来自 OCR 提取或 JSON）
    inv_data: dict,         # 商业发票结构化数据
    bl_data: dict,          # 提单结构化数据
    ins_data: dict,         # 保险单结构化数据
    presentation_date: str, # 交单日期
    config_path: str = "config/agent_config.json"
) -> EnhancedReviewResult
```

**完整流程伪代码**：

```python
def enhanced_review(lc_data, inv_data, bl_data, ins_data, presentation_date, config_path):
    # ─── 加载配置 ───
    config = load_config(config_path)
    llm = AgentLLMClient(config["llm"])
    checker = OutputChecker(DISCREPANCY_SCHEMA, config["output_checker"])

    # ─── Phase 1: 多智能体预理解 ───
    knowledge_context = ""
    if config["agents"]["planner"]["enabled"]:
        planner = PlannerAgent(llm)
        planner_result = planner.analyze(lc_data, inv_data, bl_data, ins_data)

        if config["agents"]["knowledge"]["enabled"]:
            from check_v2 import embed_model, collection  # 复用现有组件
            knowledge_agent = KnowledgeAgent(llm, embed_model, collection)
            knowledge_context = knowledge_agent.enhance(planner_result, ...)

    # ─── Phase 2: 规则引擎（复用现有） ───
    from check_v2 import check_calculations
    rule_discrepancies, compliant_items = check_calculations(
        lc_data, inv_data, bl_data, ins_data, presentation_date
    )

    # ─── Phase 3: 增强版 AI 审核 ───
    # 3a. 加载正反对比样本
    contrastive_text = ""
    if config["contrastive_examples"]["enabled"]:
        ce_manager = ContrastiveExampleManager(config["contrastive_examples"]["data_dir"])
        selected = ce_manager.select_relevant(lc_data, inv_data, bl_data, ins_data)
        contrastive_text = ce_manager.format_for_prompt(selected)

    # 3b. 构建增强版 prompt = 原始 system_prompt + knowledge_context + contrastive_text
    enhanced_system_prompt = build_enhanced_system_prompt(knowledge_context, contrastive_text)
    enhanced_user_prompt = build_enhanced_user_prompt(lc_data, inv_data, bl_data, ins_data, ...)

    # 3c. 调用 LLM + OutputChecker 校验重试
    ai_discrepancies = llm.call_json(
        enhanced_system_prompt,
        enhanced_user_prompt,
        schema=DISCREPANCY_SCHEMA,
        max_loop=config["output_checker"]["max_loop"]
    )

    # ─── Phase 4: 合并去重（复用现有） ───
    from check_v2 import merge_discrepancies
    merged = merge_discrepancies(rule_discrepancies, ai_discrepancies)

    # ─── Phase 5: 反思纠错 ───
    final_discrepancies = merged
    if config["agents"]["reflection"]["enabled"]:
        reflection = ReflectionAgent(llm, checker)
        final_discrepancies = reflection.reflect(
            merged, lc_data, inv_data, bl_data, ins_data, presentation_date
        )

    # ─── Phase 6: 构建结果 ───
    return EnhancedReviewResult(
        discrepancies=final_discrepancies,
        compliant_items=compliant_items,
        metadata={
            "planner_terms": planner_result if planner_result else None,
            "knowledge_enhanced": bool(knowledge_context),
            "contrastive_examples_used": bool(contrastive_text),
            "reflection_applied": config["agents"]["reflection"]["enabled"],
            "version": "v3_enhanced"
        }
    )
```
