## 1. 项目全景图

### 1.1 这是什么？

**信用证智能审单系统**是一个基于人工智能技术的自动化单据审核平台，专门用于处理国际贸易中的信用证（Letter of Credit, LC）单据审核工作。

#### 核心能力

| 能力维度 | 具体功能 | 技术实现 |
|---------|---------|---------|
| 📝 **单据解析** | 自动识别信用证、商业发票、提单、保险单 | PaddleOCR + 大模型字段提取 |
| 🔍 **智能审核** | 基于UCP600规则自动检测不符点 | 规则引擎 + RAG检索增强 + 大模型判断 |
| 📊 **风险评估** | 严重程度分级（严重/一般/轻微） | 多维度评分算法 |
| 📄 **报告生成** | 自动生成标准化审核报告 | 结构化模板 + 可视化展示 |

#### 处理流程概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        信用证智能审单系统                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  输入 → [OCR识别] → [AI字段提取] → [规则引擎] → [AI深度审核] → 输出  │
│         (Paddle)    (Qwen2.5)      (精确计算)   (语义理解)        │
│                            ↓                                     │

└─────────────────────────────────────────────────────────────────┘
```

### 1.2 为什么需要这个系统？

#### 行业痛点

1. **人工审核效率低**
   - 传统信用证审单需要专业人员逐条核对
   - 一套单据审核时间通常需要30-60分钟
   - 高峰期单据积压严重

2. **专业性门槛高**
   - 需要精通UCP600、ISBP745等国际惯例
   - 培养一名合格审单员需要2-3年
   - 人员流动导致经验流失

3. **错误率难以控制**
   - 人工审核易受疲劳、情绪影响
   - 复杂条款容易遗漏
   - 不一致性审核标准

4. **合规风险大**
   - 不符点判断直接影响银行拒付/承付决策
   - 错误判断可能导致法律纠纷
   - 需要完整审核留痕

#### 系统价值

| 价值维度 | 具体收益 | 量化指标 |
|---------|---------|---------|
| ⏱️ **效率提升** | 审核时间从30分钟降至3-5分钟 | 效率提升6-10倍 |
| 🎯 **准确率提升** | AI+人工复核模式降低漏检率 | 大方向判断准确率>90% |
| 💰 **成本节约** | 减少资深审单人员投入 | 人力成本降低50%+ |
| 📈 **标准化** | 统一审核标准，消除人为差异 | 一致性100% |
| 🔄 **知识沉淀** | UCP600规则库可复用、可迭代 | 知识资产化 |

### 1.3 核心技术创新

#### 三层审核架构

```
┌────────────────────────────────────────────────────────────┐
│                     三层审核架构                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌─────────────────┐                                       │

│  └────────┬────────┘                                       │
│           ↓                                                │
│  ┌─────────────────┐                                       │
│  │  第二层：AI全面审核 │ ← 语义理解（货物描述、名称一致性等）     │
│  │  ai_full_review   │   动态RAG检索 + 大模型分析               │
│  └────────┬────────┘                                       │
│           ↓                                                │
│  ┌─────────────────┐                                       │
│  │  第三层：合并去重  │ ← 消除重复检测，统一输出格式            │
│  │  merge_discrepancies                                       │
│  └─────────────────┘                                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 动态RAG检索

不同于传统的固定规则匹配，系统采用**动态检索增强生成（Dynamic RAG）**：

1. **基础查询** - 覆盖通用审核场景
2. **动态查询** - 根据单据内容实时生成（如检测到租船提单则检索Article 22）
3. **文档类型限定** - 针对不同单据类型检索相关条款

### 2.3 核心模块详解

#### 2.3.1 OCR提取模块 (`ocr_extract.py`)

```python
# 核心功能
extract_text(file_path)           # 自动识别PDF/图片，提取原始文字
extract_lc_fields(raw_text)       # AI提取信用证结构化字段
extract_invoice_fields(raw_text)  # AI提取商业发票字段
extract_bl_fields(raw_text)       # AI提取提单字段
extract_insurance_fields(raw_text)# AI提取保险单字段

# 支持的文件格式
- PDF（电子版直接提取文字 / 扫描版用OCR）
- 图片（PNG, JPG, JPEG, BMP, TIFF）
```

**提取字段示例（信用证）**：
- `lc_number`: 信用证号
- `amount`/`currency`: 金额/币种
- `applicant`/`beneficiary`: 申请人/受益人
- `latest_shipment_date`: 最迟装运日
- `loading_port`/`discharge_port`: 装卸港
- `goods_description`: 货物描述
- `partial_shipment`/`transhipment`: 分批/转运

#### 2.3.2 规则引擎 (`check_v2.py::check_calculations`)

**只做AI不擅长的精确计算**：

| 检查项 | 规则说明 | UCP条款 |
|-------|---------|--------|
| 发票金额 | 不得超过信用证金额×(1+容差) | Article 18(b) |
| 币种一致性 | 发票/保险单币种必须与LC一致 | Article 18(a), 28(f)(ii) |
| 保险金额 | 必须≥发票金额的110% | Article 28(f)(ii) |
| 装船日期 | 不得晚于最迟装运日 | Article 20(c) |
| 交单期限 | 装运日后21天内（或LC规定） | Article 14(c) |
| 信用证有效期 | 交单日期不得晚于到期日 | Article 6(d)(i) |
| 保险单日期 | 不得晚于装船日期 | Article 28(e) |

#### 2.3.3 AI审核模块 (`check_v2.py::ai_full_review`)

**处理需要语义理解的复杂场景**：

1. **货物描述一致性**
   - 发票：必须与LC"correspond with"（一致）
   - 提单：不得与LC"inconsistent"（矛盾）

2. **港口名称模糊匹配**
   - "Shanghai, China" ≈ "SHANGHAI"
   - 提取核心城市名进行匹配

3. **公司名称严格匹配**
   - 检查发票出票人 vs LC受益人
   - 检查提单托运人 vs LC受益人
   - 检查保险被保险人 vs LC受益人

4. **清洁提单判断**
   - 识别"CLEAN ON BOARD"批注
   - 检测不清洁批注（damaged, stained等）

5. **险别覆盖验证**
   - 确认保险单覆盖LC要求的全部险别
   - 检查除外条款是否与LC矛盾

#### 2.3.4 RAG知识检索 (`check_v2.py::retrieve_comprehensive_context`)

```python
# 动态查询生成逻辑
def generate_dynamic_queries(lc, invoice, bl, insurance):
    queries = []
    
    # 根据单据内容动态生成查询
    if "charter party" in bl_text:
        queries.append("租船合同提单 Article 22")
    
    if "shipper's load" in bl_text:
        queries.append("托运人装载批注 Article 26")
        
    if "on deck" in bl_text:
        queries.append("货装舱面 Article 26")
    
    if "transshipment" in bl_text:
        queries.append("转运规定 Article 20")
    
    if inv_amount > lc_amount:
        queries.append("超额发票 Article 18")
    
    return queries
```

**检索策略**：
1. 基础通用查询（确保覆盖面）
2. 动态查询（基于单据内容）
3. 文档类型限定检索（宁多勿漏）

## 5. 业务知识库

### 5.1 信用证审单基础

#### 什么是信用证？

信用证（Letter of Credit, L/C）是银行根据进口商（申请人）的要求和指示，向出口商（受益人）开立的有条件承诺付款的书面文件。

```
┌─────────────────────────────────────────────────────────┐
│                    信用证业务流程                         │
├─────────────────────────────────────────────────────────┤
│                                                          │
│   申请人(进口商)              受益人(出口商)              │
│       │                           │                      │
│       │ ① 申请开证                │                      │
│       ▼                           │                      │
│   ┌─────────┐                     │                      │
│   │ 开证行   │ ←───────────────────┤                      │
│   │ (银行)  │ ② 开立信用证         │                      │
│   └────┬────┘                     │                      │
│        │ ③ 通知                    │                      │
│        ▼                           │                      │
│   ┌─────────┐ ④ 发货              │                      │
│   │ 通知行   │◄────────────────────┤                      │
│   │ (银行)  │                     │                      │
│   └────┬────┘                     │                      │
│        │ ⑤ 交单                   │                      │
│        │ （本系统审核环节）          │                      │
│        ▼                           │                      │
│   [审单系统] → 判断是否相符 → 承付/拒付                   │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

#### UCP600核心原则

| 原则 | 含义 | 系统实现 |
|-----|------|---------|
| **独立抽象原则** | 信用证独立于基础合同 | 只审核单据，不涉货物 |
| **严格相符原则** | 单证一致、单单一致 | 发票与LC一致(correspond)，其他单据不矛盾 |
| **合理审慎原则** | 银行合理时间审核（5个银行工作日） | 系统秒级完成 |
| **表面审核原则** | 只审核单据表面，不涉及实质 | OCR+AI提取表面信息 |

### 5.2 不符点分类与严重程度

#### 严重不符点（建议拒付）

| 类型 | 示例 | UCP条款 |
|-----|------|--------|
| 金额超限 | 发票金额超过LC允许的最大值 | Article 18(b) |
| 币种不符 | 发票币种与LC不一致 | Article 18(a) |
| 超期装运 | 装船日期晚于最迟装运日 | Article 20(c) |
| 超期交单 | 超过装运后21天或LC到期日 | Article 14(c), 6(d) |
| 保险不足 | 保险金额<发票金额110% | Article 28(f)(ii) |
| 不清洁提单 | 提单有货损批注 | Article 27 |
| 港口不符 | 装卸港与LC规定不一致 | Article 20(a)(iii) |

#### 一般不符点（建议联系申请人）

| 类型 | 示例 | UCP条款 |
|-----|------|--------|
| 货物描述不完整 | 发票货物描述遗漏关键信息 | Article 18(c) |
| 保险起运地不符 | 保单起运地与提单装货港不一致 | Article 28(h) |

#### 轻微不符点（可接受）

| 类型 | 示例 | 说明 |
|-----|------|------|
| 单位表述差异 | MT vs Metric Tons | 实质相同 |
| 名称微小差异 | Co., Ltd. vs Company Limited | 不影响识别 |

### A. 故障排查

#### A.1 启动问题

| 症状 | 可能原因 | 解决方案 |
|-----|---------|---------|
| `ModuleNotFoundError` | 依赖未安装 | 运行 `pip install -r requirements.txt` |
| `Connection refused` (Ollama) | 大模型服务未启动 | 运行 `ollama serve` |
| `CUDA out of memory` | GPU显存不足 | 关闭其他程序或降低模型精度 |
| `Permission denied` | 文件权限问题 | 运行 `chmod -R 755 /root/lc-checker` |

#### A.2 审核问题

| 症状 | 可能原因 | 解决方案 |
|-----|---------|---------|
| 所有字段解析失败 | OCR识别问题 | 检查PDF质量，尝试更高分辨率 |
| AI返回格式异常 | 大模型输出不稳定 | 重试，或检查prompt模板 |
| 知识库检索为空 | ChromaDB未初始化 | 运行 `python build_knowledge_base.py` |
| 审核结果明显错误 | 模型幻觉 | 核对UCP600条款原文 |

#### A.3 性能问题

| 症状 | 可能原因 | 解决方案 |
|-----|---------|---------|
| 审核速度过慢 | GPU未启用 | 检查CUDA安装，使用`nvidia-smi` |
| 内存持续增长 | 内存泄漏 | 重启服务，检查大文件处理逻辑 |
| 并发请求卡顿 | 单线程处理 | FastAPI天然支持异步，考虑多实例部署 |
